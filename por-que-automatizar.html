SAP ↔ N8N ↔ Tookane

Por qué N8N normaliza los datos antes de enviarlos a Tookane

OData (la puerta de acceso web a SAP) deposita en PostgreSQL los mismos datos que SAP envía a Tookane hoy. Desde esa única base de datos, N8N transforma y normaliza cada dato para que llegue a Tookane exactamente como lo necesita — en el formato correcto, con el transportista identificado y con las reglas de cada carrier aplicadas.

1SAP envía los datos por partes — PostgreSQL los reúne en un único sitio
⚠ Sin N8N — SAP directo a Tookane

Hoy SAP envía datos a Tookane de forma directa, pero los datos vienen de módulos separados dentro de SAP: el transporte (fechas, ruta, transportista), el pedido de venta (referencia, cliente, importe), la dirección de entrega y los artículos. Sin una capa intermedia, Tookane recibe lo que SAP le da directamente, sin toda la información combinada y totalmente normalizada.

✓ Con N8N como intermediario

OData es el protocolo estándar de consulta de SAP — una puerta de acceso web a los datos del sistema. A través de esa puerta, los datos de los módulos de SAP se depositan en PostgreSQL, que actua como la única base de datos del proyecto. N8N lee desde ese único punto y envía a Tookane un envío completo con toda la información combinada.

2SAP y Tookane identifican a los transportistas con códigos distintos
⚠ Sin N8N — SAP directo a Tookane

SAP identifica a cada transportista con su propio código interno de proveedor (por ejemplo, un número como «1000045» que en SAP significa Dachser). Ese código llega a PostgreSQL tal cual.

✓ Con N8N como intermediario

En PostgreSQL existe una tabla con la equivalencia entre cada código de proveedor de SAP y el nombre del transportista en Tookane. N8N consulta esa tabla antes de enviar y traduce automáticamente el código al identificador correcto. Tookane recibe el carrier ya identificado, sin margen de error.

3SAP y Tookane escriben las fechas de manera distinta
⚠ Sin N8N — SAP directo a Tookane

SAP almacena las fechas en un formato propio: el 15 de junio de 2026 se guarda como «20260615», sin separadores ni indicación de hora. Ese dato llega a PostgreSQL en ese mismo formato. Tookane espera las fechas en el estándar internacional («2026-06-15»). Son el mismo dato pero escritos de forma incompatible.

✓ Con N8N como intermediario

Al leer de PostgreSQL, N8N convierte automáticamente todas las fechas del formato SAP al estándar internacional antes de enviarlas a Tookane. Así, las fechas de carga y entrega llegan siempre correctas y Tookane puede programar rutas y generar alertas sin errores.

4SAP registra el transporte desde la primera planificación, aunque aún esté incompleto
⚠ Sin N8N — SAP directo a Tookane

En SAP, el documento de transporte existe desde que se empieza a planificar, aunque todavía falten datos clave: puede que el transportista no esté asignado o que la fecha de entrega no esté confirmada. Si esos datos incompletos se enviaran a Tookane, generarían envíos erróneos o duplicados en el TMS.

✓ Con N8N como intermediario

OData solo expone a PostgreSQL los transportes que ya están completamente planificados: con transportista asignado y fechas confirmadas. N8N actua únicamente sobre esos registros maduros. Solo entonces crea el envío en Tookane, garantizando que nunca llega un transporte a medias.

5Cada transportista necesita los datos en un formato específico que SAP no conoce
⚠ Sin N8N — SAP directo a Tookane

SAP no sabe, por ejemplo, que Kuehne+Nagel necesita saber si el envío es para una empresa (B2B — entrega en almacén o empresa) o para un particular (B2C — entrega en domicilio), ni que la referencia del pedido debe ir en un campo concreto de su sistema, o que los números de cada bulto van en otro campo diferente. Sin esas reglas, los datos acaban en un cajón de sastre llamado EXTRAINFO — un campo genérico que Tookane no puede procesar automáticamente.

✓ Con N8N como intermediario

N8N tiene configuradas en PostgreSQL las reglas específicas de cada transportista. Al normalizar los datos de SAP, aplica esas reglas antes de enviar: para Kuehne+Nagel, marca el envío como B2B por defecto, coloca la referencia del cliente en el campo correcto y añade los números de bulto donde K+N los espera. Tookane recibe los datos exactamente como los necesita, sin interpretar nada.

6SAP solo tiene datos logísticos — PostgreSQL añade el contexto operativo
⚠ Sin N8N — SAP directo a Tookane

SAP tiene los datos del transporte: qué se envía, a quién y cuándo. Pero no tiene el teléfono del contacto del transportista si hay un problema, ni si ese cliente tiene acuerdos especiales de nivel de servicio, ni qué incidencias hubo antes con ese carrier en esa ruta.

✓ Con N8N como intermediario

En PostgreSQL conviven los datos de SAP con los datos operativos de CO: contactos de cada transportista, perfil de cada cliente y el historial de incidencias. Al construir el envío para Tookane, N8N combina ambas fuentes desde esa única base de datos. Tookane no solo recibe el envío — recibe el envío con todo el contexto necesario para gestionarlo bien.
