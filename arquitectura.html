Conexión SAP → N8N: opciones técnicas

Referencia para IT — comparativa de métodos y campos exactos requeridos

Ahora — puente de integración

La conexión SAP → N8N que se describe aquí es la solución operativa para poner en marcha los workflows mientras SAP BTP todavía no está operativo en Actiu. OData o RFC/BAPI nos permiten arrancar sin esperar a la plataforma de integración corporativa.

Futuro — SAP BTP + Datasphere

Cuando SAP BTP esté operativo será la capa de integración corporativa y sustituirá la conexión directa actual. Paralelamente, SAP Datasphere está planificado como plataforma de datos — cuando esté listo, la base de datos PostgreSQL actiu_co migrará a Datasphere. Por eso PostgreSQL y N8N estarán en el mismo servidor: facilita esa migración futura.

Métodos de conexión disponibles

OData / SAP Gateway★ Recomendado

HTTP/REST estándar. Actiu usa S/4HANA, donde los servicios OData están estandarizados y la cobertura de objetos de negocio (entregas, pedidos, transportes) es amplia. N8N lo consume con el nodo HTTP Request nativo — sin instalación extra. Existe además un nodo comunitario n8n-nodes-sap-odata (beta v0.5, solo OData). Autenticación Basic Auth u OAuth2. Polling periódico o delta queries por fecha/STTRG.

Sin conector adicional en N8NBasic Auth / OAuth2Polling o delta queriesS/4HANA: cobertura OData amplia
RFC / BAPIPosible con wrapper HTTP

Método fundacional de SAP. Soporte bidireccional completo y amplia cobertura. N8N no tiene nodo RFC nativo. Para integrarlo: IT expone una fachada HTTP ligera (Python/Flask o Node.js) que internamente llama a RFC/BAPI, y N8N llama a esa fachada. Alternativa válida si IT prefiere exponer objetos específicos con lógica de negocio encapsulada.

Aviso para IT

Los campos y referencias de tabla a continuación son nuestra estimación desde CO. Será IT quien tenga el conocimiento real de esas tablas y quien mejor pueda valorar de dónde extraer cada dato, si es accesible en la instalación actual y si es factible exponerlo hacia N8N. Los campos que SAP no pueda proporcionar quedarán como NULL en PostgreSQL hasta que sea posible obtenerlos — y si un campo no es factible, la base de datos se ajustará para no esperarlo. El objetivo final es normalizar los datos de SAP + Tookane + Notion + correos en un único contexto enriquecido que permita aplicar correctamente cada workflow con los resultados esperados.

Si ya existe un servicio OData de entregas/transportes activo en SAP Gateway (p. ej. API_OUTBOUND_DELIVERY_SRV), IT solo necesita activarlo y dar credenciales de lectura. N8N llama al endpoint cada hora (o con la frecuencia que se acuerde).

Opción B — CDS View personalizada ✪ Recomendado S/4HANA

IT crea una CDS View (nativa en S/4HANA) que une los campos de las tablas listadas arriba y la expone como servicio OData. Recibimos exactamente los campos que necesitamos, sin procesamiento extra en N8N. Mejor rendimiento, más fácil de mantener y preparada para la futura integración con SAP BTP.

Filtro de sincronización sugerido (STTRG): N8N filtra por STTRG = 1 (planificación completada, listo para notificar a Tookane) o por fecha de creación ≥ última sync.
