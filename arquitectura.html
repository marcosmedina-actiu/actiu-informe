Arquitectura propuesta & Workflows N8N

N8N como hub central de integración SAP ↔ Tookane

Actualizado: 10/06/2026 13:09 PostgreSQL actiu_co

2. Propuesta de automatizaciones

El objetivo es eliminar el trabajo manual repetitivo y hacer que el ciclo de vida de cada envío quede trazado automáticamente — desde que SAP lo crea hasta que se entrega al cliente.

El corazón de la propuesta: N8N como nexo central entre SAP, Tookane y PostgreSQL. SAP genera el envío → N8N lo publica en Tookane → Tookane devuelve cada cambio de estado → N8N lo guarda en PostgreSQL. Todo el ciclo queda registrado sin intervención manual.

2.1 La arquitectura propuesta

El flujo es lineal y sin intervención manual:

  1. SAP crea un envío → N8N lo lee (OData VTTK) y lo publica en Tookane con los datos correctos
  2. Tookane gestiona el envío (asigna carrier, genera tracking, coordina la entrega)
  3. Tookane avisa via webhook de cada cambio de estado → N8N lo recibe y lo guarda en PostgreSQL
  4. PostgreSQL acumula el ciclo completo: creación, asignación, salida, incidencias, entrega







¿Por qué N8N como hub central?


Una pregunta natural es si SAP o Tookane podrían hacer este trabajo internamente. No:


    

  • SAP es el sistema de registro — no está diseñado para publicar eventos en tiempo real a sistemas externos sin desarrollo ABAP
    • 

  • Tookane gestiona la logística de transporte — no habla con SAP ni con los sistemas internos de Actiu
    • 



N8N es el nexo que conecta los tres sistemas y decide qué datos van a dónde:

Ventajas de este modelo:


    

  1. Un ciclo trazado de punta a punta: desde que SAP crea el transporte hasta la entrega, cada evento queda en PostgreSQL sin intervención manual
    2. 

  2. Tookane siempre tiene los datos SAP correctos: N8N traduce el formato SAP al formato Tookane en el momento de la creación del envío
    3. 

  3. Coste de cambio mínimo: modificar la lógica de integración cuesta horas en N8N, no semanas en SAP o Tookane
    4. 

  4. Sin riesgo para el ERP: ningún sistema externo escribe en SAP
    5.

3. Detalle de cada workflow




Los ocho workflows no son piezas independientes — se alimentan entre sí para resolver
de forma coordinada múltiples problemas a la vez. Cada uno aporta datos que el siguiente
necesita, creando un ciclo que unifica la información y mejora la trazabilidad operativa.





📡



Workflow W1


SAP → PostgreSQL → Tookane (campos correctamente mapeados)












¿Cuándo se activa?


Cuando SAP crea o modifica una orden de transporte






¿Quién se beneficia?


Tookane (puede aplicar reglas automáticas) · IT · Equipo logística








¿QUÉ HACE PASO A PASO?


    

  1. SAP crea o modifica un transporte (orden VTTK)
    2. 

  2. SAP emite un evento automático que N8N recibe
    3. 

  3. N8N transforma los datos: convierte Z000 → STANDARD, Z001 → EXPRESS, etc.
    4. 

  4. N8N llama a la API de Tookane con los campos correctos (serviceType, agreedDeliveryDate, etc.)
    5. 

  5. Tookane recibe la expedición con toda la información estructurada y puede actuar automáticamente
    6. 

  6. N8N guarda el registro en PostgreSQL para trazabilidad completa
    7. 









HOY (sin workflow)


Datos SAP en EXTRAINFO genérico; Tookane no puede aplicar reglas ni automatizaciones sobre ellos






CON WORKFLOW


Datos en campos dedicados; Tookane asigna carrier y crea alertas automáticamente













🔄



Workflow W2


Tookane → PostgreSQL (sincronización en tiempo real)












¿Cuándo se activa?


Cuando Tookane cambia el estado de cualquier expedición (entrega confirmada, incidencia detectada, etc.)






¿Quién se beneficia?


Equipo CO · Técnicos comerciales · Dirección








¿QUÉ HACE PASO A PASO?


    

  1. Tookane detecta un cambio de estado (por ejemplo, una entrega completada)
    2. 

  2. Tookane envía una notificación automática a N8N (en menos de un minuto)
    3. 

  3. N8N recibe la notificación y extrae los datos relevantes
    4. 

  4. N8N actualiza PostgreSQL con el nuevo estado de la expedición
    5. 

  5. Si el cambio es relevante (incidencia, retraso), N8N activa otros workflows
    6. 









HOY (sin workflow)


Estado de las entregas actualizado 1-2 veces por semana, mediante descarga manual de CSV






CON WORKFLOW


Estado actualizado en menos de 1 minuto, sin intervención humana













📊



Workflow W3


Google Sheets de seguimiento (automático)












¿Cuándo se activa?


Cada hora, automáticamente






¿Quién se beneficia?


Técnicos comerciales · Equipo CO · Cualquier persona que use el Excel actual








¿QUÉ HACE PASO A PASO?


    

  1. N8N se activa cada hora de forma programada
    2. 

  2. Consulta PostgreSQL: expediciones activas, estados, ETAs, alertas
    3. 

  3. Calcula automáticamente qué expediciones tienen la ETA vencida (semáforo rojo)
    4. 

  4. Escribe los datos en Google Sheets, organizados por carrier y fecha
    5. 

  5. El técnico comercial abre el Sheets y ve todo actualizado, sin hacer nada
    6. 









HOY (sin workflow)


30-45 minutos diarios por técnico descargando CSV, pegando datos en Excel y actualizando colores manualmente






CON WORKFLOW


0 minutos de trabajo manual · Datos en tiempo real · Semáforo de alertas automático













⏰



Workflow W4


Alerta proactiva de ETA vencida












¿Cuándo se activa?


Cada mañana a las 7:00, automáticamente






¿Quién se beneficia?


Equipo CO · Técnicos comerciales








¿QUÉ HACE PASO A PASO?


    

  1. A las 7:00, N8N consulta PostgreSQL: ¿qué expediciones tienen ETA de ayer o antes y no están entregadas?
    2. 

  2. Agrupa por transportista: Dachser (X vencidas), DSV (Y vencidas)...
    3. 

  3. Genera un email con la lista de vencidas, el contacto del carrier y el histórico de esa ruta
    4. 

  4. El equipo CO recibe el email antes de que empiece la jornada
    5. 

  5. Pueden actuar proactivamente, antes de que el cliente llame preguntando por su pedido
    6. 









HOY (sin workflow)


El equipo se entera de los retrasos cuando el cliente llama · Revisión manual de ETA vencidas lleva 45-60 min/día






CON WORKFLOW


Resumen automático cada mañana con todo listo para actuar · 0 min de revisión manual

Workflow W5


Incidencia en Tookane → Caso CO automático












¿Cuándo se activa?


Cuando Tookane marca una expedición como «Incidencia»






¿Quién se beneficia?


Equipo CO · Cliente final (recibe respuesta más rápida)








¿QUÉ HACE PASO A PASO?


    

  1. Tookane detecta una incidencia (rotura, dirección incorrecta, imposible entrega...)
    2. 

  2. Tookane avisa a N8N mediante webhook en menos de 1 minuto
    3. 

  3. N8N busca en PostgreSQL toda la información de esa expedición (pedido, cliente, historial)
    4. 

  4. N8N clasifica el tipo de incidencia (según patrones aprendidos)
    5. 

  5. Se crea un caso CO con toda la información ya cargada
    6. 

  6. El técnico CO asignado recibe una notificación con el contexto completo y el SOP a seguir
    7. 

  7. El técnico puede responder al cliente con información precisa desde el primer momento
    8. 









HOY (sin workflow)


El equipo se entera de la incidencia cuando el cliente escribe o llama · Lag típico: 24-72 horas






CON WORKFLOW


Caso creado automáticamente en < 5 minutos · El técnico actúa antes de que el cliente lo note













🤖



Workflow W6


CO Email Agent + Asignación inteligente












¿Cuándo se activa?


Cuando llega un correo nuevo al buzón de Customer Operations






¿Quién se beneficia?


Equipo CO · Cliente final (respuesta más rápida y precisa)








¿QUÉ HACE PASO A PASO?


    

  1. Llega un email al buzón CO de Gmail
    2. 

  2. N8N extrae asunto, remitente, cuerpo del email y cualquier referencia de pedido
    3. 

  3. Un agente IA (Gemini Flash) recibe el email con contexto completo preparado
    4. 

  4. El agente consulta PostgreSQL: ¿el asunto coincide con algún patrón de caso aprendido? (tipo de caso + confianza)
    5. 

  5. El agente consulta PostgreSQL: ¿qué técnico CO está disponible hoy con menor carga de casos?
    6. 

  6. Con esa información el agente clasifica el tipo de caso y redacta el borrador de respuesta
    7. 

  7. N8N registra el caso, asigna el técnico y guarda el borrador en PostgreSQL
    8. 

  8. El técnico CO recibe el email ya clasificado con el contexto logístico completo y el borrador listo
    9. 

  9. El técnico revisa, ajusta si es necesario y envía con un clic
    10. 









HOY (sin workflow)


15-25 minutos por correo buscando información en Tookane y SAP para responder con datos correctos






CON WORKFLOW


2-3 minutos por correo · Respuesta con datos precisos en tiempo real · Caso registrado y técnico asignado automáticamente

Workflow W7


Google Calendar → Disponibilidad de técnicos












¿Cuándo se activa?


Cada día a las 06:30, automáticamente






¿Quién se beneficia?


Equipo CO · Sistema de asignación automática W6








¿QUÉ HACE PASO A PASO?


    

  1. A las 06:30, N8N consulta Google Calendar de cada técnico CO
    2. 

  2. Detecta si el técnico tiene eventos de ausencia (vacaciones, formación, jornada reducida)
    3. 

  3. Calcula las horas disponibles según la agenda del día
    4. 

  4. Actualiza la tabla disponibilidad_tecnicos en PostgreSQL
    5. 

  5. W6 ya puede consultar disponibilidad real antes de asignar el próximo caso
    6. 









HOY (sin workflow)


Asignaciones sin tener en cuenta si el técnico está de vacaciones o con jornada reducida






CON WORKFLOW


Asignación automática respeta disponibilidad real · 0 casos asignados a técnicos ausentes













📖



Workflow W8


Notion → Procedimientos CO (sincronización completa)












¿Cuándo se activa?


Manual (primer arranque o cuando sea necesario) o automáticamente cada mañana a las 07:00






¿Quién se beneficia?


Equipo CO · Sistema de clasificación de casos W5 y W6








¿QUÉ HACE PASO A PASO?


    

  1. Se ejecuta manualmente o a las 07:00 cada día
    2. 

  2. Consulta la BD Notion: obtiene todos los procedimientos con «Fecha de aprobación» definida (hoy: 11 procedimientos)
    3. 

  3. Filtra las plantillas — solo sincroniza procedimientos reales
    4. 

  4. Para cada procedimiento: extrae código, nombre, tipo de caso, prioridad, SLA, palabras clave y tipo de remitente
    5. 

  5. Descarga el contenido completo (bloques Notion → texto estructurado con jerarquía de headings y listas)
    6. 

  6. Inserta en PostgreSQL los procedimientos nuevos · Actualiza los ya existentes si cambiaron
    7. 

  7. W5 y W6 utilizan automáticamente el procedimiento actualizado al clasificar el próximo caso
    8. 









HOY (sin workflow)


SOPs desactualizados en el sistema · Técnicos buscan el procedimiento correcto manualmente cada vez






CON WORKFLOW


11 procedimientos CO siempre sincronizados · El caso incluye el SOP vigente desde el primer momento · 0 búsquedas manuales
